# Day 4: Manual Test Cases for Action Extraction & Auto Replies

Welcome to Day 4 testing! Since we implemented the logic backend in `services`, we need to simulate real-world emails in your inbox, and then let your program process them.

## Prerequisites
- Make sure Ollama is running (`llama3.2` model) locally.
- Make sure your MongoDB is running.
- Ensure your Gmail account is authenticated (you can use your own account to send emails to yourself).

---

### Test Case 1: Standard Action Extraction
**Goal**: Verify that the AI parses tasks correctly and logs them as `ActionItem`s.

1. **Setup**: Open Gmail and send an email to yourself with the following:
   - **Subject**: Website Redesign Tasks
   - **Body**: "Please ensure the new hero image is uploaded by tomorrow. Also, fix the broken navigation link."
2. **Execute**: Run your sync loop. (If `terminalChat.js` initializes your heartbeat or if you have an `index.js` that pulls emails, run that script: `node server/index.js` or `node server/terminalChat.js`).
3. **Verify**:
   - Check your MongoDB instance (e.g., using MongoDB Compass). Look inside the `ActionItem` collection.
   - You should see two pending action items. 
   - `source_email` should match the Thread ID of the email you just sent.

---

### Test Case 2: PII Privacy Guard
**Goal**: Verify that the AI filters out PII (Personally Identifiable Information).

1. **Setup**: Send an email to yourself with the following:
   - **Subject**: Urgent Onboarding
   - **Body**: "Please contact John Doe at 555-0198 and email him the documents at john.doe@example.com ASAP."
2. **Execute**: Run the backend sync loop again so it ingests the new email.
3. **Verify**:
   - Check MongoDB for the generated `ActionItem`.
   - The `action` and `owner` fields must **NOT** contain "John Doe", "555-0198", or "john.doe@example.com". They should have been replaced with generic terms (e.g., "Contact the client and send the documents").

---

### Test Case 3: FYI Email Detection & Draft Creation
**Goal**: Verify the system detects an informational email, prints an `[ALERT]`, and safely creates a draft.

1. **Setup**: Send an email to yourself with the following:
   - **Subject**: Confirming 5 PM Call
   - **Body**: "Just wanted to let you know I will be attending the 5:00 PM call today."
2. **Execute**: Run the backend sync loop.
3. **Verify**:
   - **Terminal Output**: Look at your console running your app. You should see an alert:
     ```
     [ALERT] A new FYI email was received.
     From: <Your Email>
     Subject: Confirming 5 PM Call
     I am going to respond in the following way:
     <Draft Reply from AI>
     ```
   - **Gmail Output**: Go to your actual Gmail "Drafts" folder. You should see a draft addressed back to yourself containing the auto-generated response, with a subject like "Re: Confirming 5 PM Call".

---

### Test Case 4: Non-Informational Email (Negative Test)
**Goal**: Verify the system doesn't create drafts for emails that aren't purely informational or action-required.

1. **Setup**: Send an email to yourself with:
   - **Subject**: Personal Update
   - **Body**: "Hey, let's grab coffee this weekend!"
2. **Execute**: Run the backend processing loop.
3. **Verify**: 
   - The email should ideally be classified as `personal` or `meeting-related`.
   - No `[ALERT]` should appear in the console.
   - No draft should be created in your Gmail account.
